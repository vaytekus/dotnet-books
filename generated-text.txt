Результати та висновки
                                   
Загальна картина
Усі фільтри швидші за baseline (no filter), що логічно — менше рядків повертається → менше об'єктів матеріалізується EF Core. 

Ключові спостереження
 
1. Текстові фільтри схожі між собою (~0.47x)
- ByTitle, ByAuthor, ByGenre — майже однаковий час (~956 µs) і пам'ять (~460-490 KB)
- Contains() по рядку дає однакову вартість незалежно від того, чи це пряме поле чи через навігаційну властивість (Include)

2. Числові/датові фільтри швидші (~0.30x)

- ByPages і ByDate — ~620 µs, ~320 KB
- Числові порівняння (>=, <=) ефективніші за LIKE в SQLite         

3. Combined — найшвидший (0.23x)
- Не тому що "розумніший", а тому що повертає найменше рядків → менше об'єктів у пам'яті
- Це важливо: EF Core матеріалізує кожну Book разом з Author, Genre, Publisher (через Include)   

4. Пам'ять прямо пропорційна кількості результатів
- 1119 KB (all) → 206 KB (combined) — різниця ~5x лише за рахунок кількості книг  

Основний "винуватець" пам'яті — Include(Author).Include(Genre).Include(Publisher) для кожного запиту. Навіть якщо потрібен лише заголовок — завантажуються всі зв'язані сутності.

Потенційна оптимізація для майбутнього бенчмарку: AsNoTracking() — прибирає overhead трекінгу змін і може зменшити алокації на ~15-20%.

AsNoTracking() погіршив показники. Порівняємо:

┌─────────────────┬────────────────────┬──────────────────────┐                                                        
│                 │   До (tracking)    │ Після (AsNoTracking) │
├─────────────────┼────────────────────┼──────────────────────┤                                                        
│ No filter       │ 2,024 µs / 1119 KB │ 4,185 µs / 3156 KB   │                                                        
├─────────────────┼────────────────────┼──────────────────────┤                                                        
│ Filter by title │ 956 µs / 490 KB    │ 1,959 µs / 1364 KB   │                                                        
└─────────────────┴────────────────────┴──────────────────────┘                                                        
                                   
Identity map — ключова деталь. З tracking EF Core зберігає кожен об'єкт за ID в кеші. 1000 книг → 5 авторів, 5 жанрів, 3 видавці — всі книги посилаються на одні й ті ж об'єкти в пам'яті.               
              
З AsNoTracking() identity map вимкнений — EF Core створює новий об'єкт Author/Genre/Publisher для кожної книги окремо. 1000 книг × 3 Include = ~3000 зайвих об'єктів.                                    
              
Висновок                             
              
AsNoTracking() вигідний коли:        
- Дані унікальні (немає повторюваних навігаційних властивостей)
- Використовується Select() проекція замість Include()                                                                 
                                                    
У вашому випадку (багато книг → мало авторів/жанрів) — tracking корисніший, він економить пам'ять через дедуплікацію.
